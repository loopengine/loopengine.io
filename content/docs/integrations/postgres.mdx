**Archetype:** [Store](/docs/concepts/integration-adapter#store-adapter) — implements `LoopStore`, slotted at `options.store`.

## What it does

Swaps the in-memory adapter for PostgreSQL persistence. Loop state, transitions, and events are stored in Postgres through the same `LoopStore` interface, so loop definitions and business logic remain unchanged.

## Quick setup

1. **Install**

   ```bash
   npm install @loop-engine/adapter-postgres
   ```

2. **Create a `LoopStore` with `postgresStore`**

   ```typescript
   // @no-typecheck
   import { createPool, postgresStore } from "@loop-engine/adapter-postgres";
   import { createLoopSystem } from "@loop-engine/sdk";

   const pool = createPool({ connectionString: process.env.DATABASE_URL! });
   const store = postgresStore(pool);
   const { engine } = await createLoopSystem({
     loops: [procurementLoop],
     store
   });
   ```

3. **Local vs production pattern**

   ```typescript
   // Dev: in-memory (default) — omit `store`
   const dev = await createLoopSystem({ loops: [procurementLoop] });

   // Prod: pass `store: postgresStore(pool)` as above
   ```

## Schema note

On first startup, the adapter creates required persistence tables:

- `loop_instances`
- `loop_events`
- `loop_transitions`

## Configuration reference

| Option | Type | Default | Description |
|---|---|---|---|
| `connectionString` | `string` | required | Postgres connection URL |
| `maxConnections` | `number` | `10` | Connection pool size |
| `ssl` | `boolean` | `false` | Enable SSL for connection |

## Troubleshooting

- **Connection refused** — Verify database host/port and credentials.
- **Migration failed** — Adapter user requires `CREATE TABLE` privileges.
- **Slow queries** — Add targeted indexes for your query paths, for example:

  ```sql
  CREATE INDEX IF NOT EXISTS idx_loop_instances_loop_id
  ON loop_instances(loop_id);
  ```
